What does each runtime feature stand for?


You can find different Runtime Features in RCPTT Advanced Options. Here you can find the description of the most of them.

  • Logging
    1. Enable Activity Logging: expanded activities logging
    2. Enable per command image capture: when enabled, should take a screenshot on each ECL command

  • Reporting
    1. Include ignored timers: if enabled, include in Report all timerExecs info even those which we are not waiting for.
    2. Include wait details: when enabled, includes wait details into Report
    3. Include trace and take screenshot: if enabled, screenshots made by 'take-screenshot' ECL command are included in Execution Report.

  • Runtime Limits
    1. Test Execution Timeout: Timeout for a single test execution in seconds
    2. Context operation runnable timeout: Context applying timeout.
    3. Time RCPTT wait for AUT application to startup: How many seconds RCPTT should wait for application startup.
    4. AutoBuild Job Hang Skip timeout: how long wait for AutoBuild (usually, when we import resources into AUT workspace)
    5. Log waiting for job if exceed: if job takes more then a limit, RCPTT writes to log that it waits for a job
    6. Job Hang Skip timeout: Job hang skip timeout in milliseconds. If job is running longer than this time, RCPTT Runtime considers that it is hung and moves to the next command.
    7. Enable step mode for job after timeout: If job executes more than this time (in milliseconds) and sleeps (i.e. executing Thread.sleep() or Object.wait()), then RCPTT considers that this job is waiting for some user actions and continues to the next command.
    8. Step mode: One step every: When job is in sleeping mode (see jobTreatAsSleeingTimeout option), execute commands with given delay (in milliseconds) between commands.
    9. Step mode: Step mode timeout: Wait timeout in milliseconds for stepping jobs (see jobTreatAsSleepingTimeout and jobSleepingStepTime)
    10. Wait for delayed jobs with delay less then: Max job scheduled delay to be waited for in milliseconds. If job is scheduled with delay less than this value, RCPTT sets delay to 0 and waits for job completion (also see jobNullifyRescheduleMaxValue). Otherwise RCPTT Runtime does not wait for job completion if it is scheduled with a delay greater than this value.
    11. Wait jobs completion after context apply:If there are any jobs started after context applying, wait for their completion during this time (in milliseconds).
    12. Delayed jobs scheduled with delay 0, if not rescheduled more then: If job reschedules itself more than times specified by this parameter, RCPTT stops setting delay to 0 (see jobScheduleDelayedMaxtime parameter).
    13. Timeout for background Debug plugin launch Job: Timeout in milliseconds for jobs scheduled from eclipse Debug plugin
    14. Nullify timerExec if time is less then: If Display.timerExec is scheduled for the delay less than this value (in milliseconds), set delay to 0.
